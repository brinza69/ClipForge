## 2. How to operate the rig (it's autonomous now)

**You normally don't need to do anything** — it runs on its own and resumes from
the last unprocessed sheet row. But for reference:

| Action | Command / where |
|---|---|
| Watch live | `http://localhost:8420/exports/live.html` |
| Watchdog log | `data/watchdog.log` |
| Dispatcher log (row→job) | `data/dispatch.log` |
| Start watchdog by hand | `powershell -ExecutionPolicy Bypass -File scripts\watchdog.ps1` |
| Cold-start rig only (no supervisor) | `.\scripts\start-dual-gpu.ps1` |
| Stop everything | end the watchdog process, then kill the `python.exe` backends/dispatcher |
| Disable logon auto-start | remove HKCU `...\Run\ClipForgeWatchdog` |

**How "resume from where it left off" works:** the dispatcher
(`scripts/dual_dispatch.py`) reads the sheet, and processes any row where
`row >= MIN_ROW` (126), the URL column (B) starts with `http`, AND the
description column (D) is empty. It assigns one row per free GPU, and on
completion writes the AI description to column D. So a row is "done" only once D
is filled — it will never re-pick a completed row, and it always continues from
the first still-empty row. **No credits from me are needed for this loop.**

> Minor known tradeoff: if a row crashed *mid-way* (D still empty), it gets fully
> re-done on resume, which can re-upload the variants that had uploaded before the
> crash. Acceptable for now; not worth de-duping.

### Status — RESEARCH DONE (full cited report: `RESEARCH-antiban-posting-2026-06-25.md`)
Verdict (verified):
- **Feasible: YES.** One TikTok app can Direct-Post to 10–50+ accounts (per-user
  OAuth, no per-account app); one Meta app can manage 10–50+ **Pages** (per-Page /
  System-User tokens). Both via official APIs only.
- **#1 blocker = the TikTok Direct Post app audit** (~2–4 wks, reviews a compliant
  posting UI). Until passed, posts are forced **SELF_ONLY (private)** + 5 users/24h.
  **Avoid it by using an aggregator that already holds an audited app.** 2nd blocker:
  Meta App Review + Business Verification for `pages_manage_posts` (Pages you don't own).
- **Facebook = Pages ONLY** — the API cannot post to personal profiles. Any
  "FB account" must become a Page. Reels cap = 30/Page/24h (fine).
- **Multi-country reality (corrected):** account region is fixed at **creation +
  human login**, NOT at API-post time — a foreign account must be *born* on that
  country's residential/mobile IP. The posting API call's IP is not the main signal,
  BUT do **not** blast the whole fleet from one shared/datacenter IP (account-linking
  trigger). Per-account proxies/antidetect browsers matter **only** at the
  creation/OAuth-login step, not for API posting.
- **Caption uniqueness:** vary the RO description per account (reusing one verbatim
  caption across the fleet is a spam-cluster signal). Map distinct variants→accounts.

### Recommendation (from the report)
- **Pilot now:** n8n (local, orchestrator only) → **Upload-Post** (cheapest SaaS,
  official APIs, bundled n8n node, inherits the audits) for the RO-native slice.
- **Scale + foreign:** graduate to **self-hosted Postiz, one instance per country**
  behind that country's residential/mobile proxy (the clean way to honor multi-country
  on official APIs). Ayrshare Business is the purpose-built-but-pricey alt (~$1,230/mo
  @100 profiles). Blotato is RO-friendly but caps ~40 accounts. Skip Mixpost (audit
  gated behind $1,199 Enterprise).
- **n8n's job:** Sheets trigger → account-map lookup (a new sheet tab: role+country→
  profile key) → Drive direct-download URL → fan-out 1 API call per variant×account →
  write result back to the sheet. The aggregator owns OAuth/audits/token-refresh.
